mkdir -p ~/aulas/dados

- To return to a parental directory, use the following command:

```shell
cd ..
```

> *Notice the space between 'cd' and '..'*

If you intend to navigate two directory levels at once, use the command:

```shell
cd ../..
```

- Let's change to the root directory and return to the working directory:

```shell
cd /
cd /home
cd /bioinfo
```

```shell
cd /home/bioinfo
```

- In this case, the placement of the **/** is crucial. Look at the two command examples below (do not execute them):

```shell
cd /step1/step2
cd step1/step2/
```

The first command is an absolute path. It instructs the machine to go to the root directory (**/**), then go to the ```step1``` directory and then to the ```step2``` directory which is inside ```step1```. In this case only a single ```/step1/step2``` directory could exist on this machine. The second command specifies a relative path. It instructs the machine that from the current location, it goes to the ```step1``` directory and then to the ```step2``` directory. There could be other ```step1/step2``` directories but inside other directories.

- Using the ```cd ..``` command allows us to change directories relative to where we currently are. You can move to a directory based on its absolute location.

- When you are in your *home* directory, instead of the directory name before the **$**, we will have the character ```~```. This is because Unix uses **~** as a shorthand to represent the user's *home*. See what happens when you use the following commands (use the ```pwd``` command after each one to confirm the results):

```shell
cd /
cd ~
cd
```

- You will notice that ```cd``` and ```cd ~``` do the same thing: both take you to your *home* directory, from wherever you are in the machine's structure. Using ```cd``` is a fast way to get there.
- You can also use the **~** character as a fast way to navigate the subdirectories of your *home*, when you are in any other:

```shell
cd ~/aulas/dados
```

pwd
```

or

pwd
```

in both, the output should be the same:

```shell
/home/bioinfo/tmp
```

## Working with files

From now on, we will work with Unix commands that are directly related to working with files. Before that, we have to create some files. The ```touch``` command will allow us to create new, empty files (it does other things too, but for now we only need files).

```shell
cd aulas
touch jedi.txt
touch federal.txt
ls
```

You will then verify the two files.

- Now let's assume that we want to move these files to a new directory, *temp*. Let's now use the ```mv``` command (do not forget to use ```tab``` to complete from now on):

```shell
mkdir temp
mv jedi.txt temp
mv federal.txt temp
ls temp
```

For the ```mv``` command we always have to specify a source file or directory (*source*) that we want to move, and then specify the target location, the destination location. We could have moved both files at once, using any of the commands below:

```shell
mv *.txt temp 
mv *t temp 
mv *ed* temp
```

The ```*``` is a wildcard, which means *any pattern it has*. The ```?``` character is another wildcard, with a slightly different meaning. See what it means.

- The ```mv``` command can also rename files. Do the example below:

```shell
touch samurai
ls
mv samurai temp/ninja
ls temp/
```

Having as result:

```shell
jedi.txt federal.txt ninja
```

The logical extension of this is that you can also use the ```mv``` command to rename a file without moving it. To rename you can also use the ```rename``` command (see its manual for options).

- It is important that you understand that as long as you describe in a command a **source** and a **destination** directory when moving a file, then it doesn't matter which directory you are in. Moving directories is similar to moving files:

```shell
mv temp temp2
ls temp2
```

- Removing directories that have files is a step that must be performed with extreme care. This is because there is no going back in many cases. The ```rm``` command is extremely dangerous for exactly this reason. You can delete everything with this command, including the *home* directory.

> Read carefully the use of the ```rm``` command to avoid much *death and destruction of the world as we know it*.

- Fortunately there is a way to make the ```rm``` command a little safer: using the ```-i``` option, which will ask for confirmation before execution. Make a habit of always using ```rm -i```. 

```shell
cd temp
ls
rm -i jedi.txt federal.txt ninja
```

which will ask at each step, which you confirm by pressing ```y```:

```shell
rm: remove regular empty file 'jedi.txt'? y
rm: remove regular empty file 'federal.txt'? y
rm: remove regular empty file 'ninja'? y”
```

This command could also have been simplified using some of the *wildcards* learned above or more complex deleting each file with a different command.

- The ```cp``` command is used for copying files and directories and has a similar syntax to the ```mv``` command, however it leaves the file at the *source*. Let's create a new file and then make a copy:

```shell
touch file1
cp file1 file2
ls
```

And to copy a file to a different directory:

```shell
touch ~/aulas/file3
cp ~/aulas/file3 ~/aulas/tutoriais/
```

For copying directories, the ```cp``` command is used with the ```-R``` or ```-r``` options. Check in ```man cp``` the difference between them.

## Viewing and editing file content

Two useful commands to view the content of files are: ```more``` or ```less```. They do not edit the content. We will use the first command we learned, ```echo``` to put text in a file and then visualize it:

```shell
echo "Call me Ishmael."
Call me Ishmael.
echo "Call me Ishmael." > primeira_linha.txt
ls
```

giving:

```shell
primeira_linha.txt
```

The ```>``` character when inserted at the end of a command redirects the *print* of that command to a file. In this case, we are putting the result of the ```echo``` command into a file ```primeira_linha.txt```. 

> *Be very careful when using redirection with ```>```: it will overwrite any file that already exists with the name that was used.*

We can now see the contents of the file:

```shell
more primeira_linha.txt
```

When using ```more``` or ```less```, you can access a help command page by pressing ```h```, pass a page using ```space```, navigate line by line using ```j``` or ```k```, ```b``` to go to the previous page and ```q``` to exit. These commands also do a million other things, including text search, but we will see that later.

The ```cat``` command is one of the most used commands when we are working with texts and characters in the terminal. Let's go to a quick demonstration, adding a new line to the previous file:

```shell
echo "The primroses were over." >> primeira_linha.txt
cat primeira_linha.txt”
```

The result is:

```shell
Call me Ishmael.
The primroses were over.
```

Pay close attention that now what was used was ```>>``` and not just ```>```. This operator appends to a file. If ```>``` had been used, the file primeira_linha.txt would have been overwritten.

The ```cat``` command only shows the content of a file and returns it to the command line. Unlike ```less``` you have no control over the display. You can use ```cat``` to quickly combine multiple files or even copy one:

```shell
cat primeira_linha.txt > primeira_linha_copia.txt
```

Many times we need information about a file without having to view it with ```more``` or ```less```. An example of this is when we need to count characters in a file. For this, a widely used command is ```wc``` (*word count*):

```shell
wc primeira_linha.txt
2  7 42 primeira_linha.txt

$ wc -l primeira_linha.txt
2 primeira_linha.txt
```

This command returns the number of lines, words, and characters in a specific file and you can use command line options to see only one of these statistics (case of ```wc -l```).

## The ```grep``` command

Use nano to create a file called `texto_teste.txt`, which will contain the following lines (no need to type, you can copy and paste):

```shell
It is a period of civil war. Rebel spaceships, striking from a hidden base, have won their first victory against the evil Galactic Empire.

```

Working on the command line, on various occasions you will have to look for patterns within files. An example in bioinformatics is searching for the number of sequences in a given fasta file (we will see later) or the number of ```ATG``` (which is the start codon) in a given sequence. The Unix ```grep``` command does this (and much, much more). The examples below show how you can use the command line options of ```grep``` to:

- Show lines that have a certain pattern.
- Ignore uppercase and lowercase when searching (```-i```)
- Only search for whole words (```-w```).
- Show the lines that DO NOT have a pattern (```-v```).
- Use *wildcards* and other patterns to allow alternatives (```*```, ```.``` and ```[]```). Let's go:

Lines containing the word 'plans':

```shell
$ grep plans texto_teste.txt
```

Highlight the found word:

```shell
$ grep --color=AUTO plans texto_teste.txt
```

Show the lines that do not have the pattern 'plans':

```shell
$ grep -v plans texto_teste.txt
```

## Combining commands with *pipes*

One of the most powerful features of Unix is that you can send the output of one program to another command (provided the other program accepts it), using something known as a ```pipe```. This is implemented using the "|" character. It's like connecting two programs. Let's see some examples:

```shell
grep plans texto_teste.txt | wc -c
    310
```

Remember the `history` command? You can search among the commands you typed using a pipe with `grep`:

```shell
history | grep mkdir
```

With the command above you extracted from the history of your commands only the lines that had the `mkdir` command.

## The environment variable `$PATH`

Another use of the `echo` command is to show the content of environment variables (*environment variables*). These contain user-specific or system-specific values that either reflect information or list useful locations in the file system. Some examples:

```shell
$ echo $USER
bioinfo
$ echo $HOME
/home/bioinfo
$ echo $PATH
/usr/local/sbin:/usr/local/bin:/home/bioinfo/bin”
```

The last one shows the content of the `$PATH` environment variable, which shows the list of directories where the programs you will execute (the executables) are expected to be. This includes all Unix commands used so far. Knowing how to change your `$PATH` to include custom directories is often necessary (for example, when you need to install a new bioinformatics tool in a non-standard location).
